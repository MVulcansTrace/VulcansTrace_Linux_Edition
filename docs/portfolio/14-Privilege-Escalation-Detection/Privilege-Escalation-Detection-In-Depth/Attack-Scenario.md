# Attack Scenario: Privilege Escalation Detection

## The Security Problem

An attacker has gained a foothold on a compromised workstation (192.168.1.150) inside a corporate network and wants to escalate privileges by compromising an administrative service on the production Linux server (10.0.0.50). The attacker tries two strategies: first, a brute-force SSH attack to gain shell access, and second, a multi-port sweep to find the weakest admin service.

---

Consider the following iptables log entries generated by an SSH brute-force attack from `192.168.1.150` against `10.0.0.50`:

kernel: Feb 14 14:00:11 server IN=eth0 SRC=192.168.1.150 DST=10.0.0.50 PROTO=TCP SPT=41004 DPT=22

## Detection Walkthrough: Admin Spike

| Step | Operation | Result |
|---|---|---|
| 1. Guard | `EnablePrivilegeEscalationDetection = true`, events.Count = 7 | Proceed |
| 2. Build admin port set | Baseline {22, 2222, 2200, 22022, 3389, 5900, 5432, 3306} | 8 ports |
| 3. Filter | All 7 events target port 22 (in admin set) | 7 events pass |
| 4. Group by source | All 7 events share `SRC=192.168.1.150` | One group |
| 5. Order by timestamp | Already sequential | 14:00:05 → 14:00:17 |
| 6. Spike detection | All 7 events fall within a 12-second span inside the 5-min sliding window | 7 events >= 5 threshold |
| 7. Threshold check | 7 >= 5 (spike threshold) | Finding triggered — **High severity** |

## Worked Example: Admin Port Sweep

Now consider the same attacker performing a multi-port admin service sweep:

```
kernel: Feb 14 14:30:01 server IN=eth0 SRC=192.168.1.150 DST=10.0.0.50 PROTO=TCP SPT=42001 DPT=22
kernel: Feb 14 14:30:03 server IN=eth0 SRC=192.168.1.150 DST=10.0.0.50 PROTO=TCP SPT=42002 DPT=3389
kernel: Feb 14 14:30:05 server IN=eth0 SRC=192.168.1.150 DST=10.0.0.50 PROTO=TCP SPT=42003 DPT=5900
kernel: Feb 14 14:30:07 server IN=eth0 SRC=192.168.1.150 DST=10.0.0.50 PROTO=TCP SPT=42004 DPT=5432
```

---

## Detection Walkthrough: Admin Port Sweep

| Step | Operation | Result |
|---|---|---|
| 1. Guard | `EnablePrivilegeEscalationDetection = true`, events.Count = 4 | Proceed |
| 2. Build admin port set | Baseline ports (22, 3389, 5900, 5432 all included) | 8 ports |
| 3. Filter | All 4 events target admin ports | 4 events pass |
| 4. Group by source | All 4 events share `SRC=192.168.1.150` | One group |
| 5. Order by timestamp | Already sequential | 14:30:01 → 14:30:07 |
| 6. Spike detection | 4 events in 14:30–14:34 window | 4 < 5 → no spike finding |
| 7. Sweep detection (i=0) | Window 14:30:01–14:35:01 contains 4 events, 4 distinct ports | 4 >= 3 → **finding triggered — Medium severity** |

## Design Rationale

The two scenarios demonstrate how the dual sub-detectors cover complementary attack strategies:

**Scenario 1 (Brute Force):** The attacker targets only SSH (port 22) with 7 rapid attempts in 12 seconds. The spike detector fires (7 >= 5) at High severity because a successful brute-force compromise would give the attacker immediate shell access. The sweep detector does not fire because only 1 distinct port is involved.

**Scenario 2 (Service Enumeration):** The attacker probes 4 different admin services in 6 seconds, with only 1 attempt per service. The spike detector does not fire (4 < 5) because the total attempt count is below the brute-force threshold. The sweep detector fires (4 distinct ports >= 3) at Medium severity because the pattern indicates reconnaissance across administrative interfaces.

If the **High** profile were used (10-minute window), both attacks would still be detected because all events in each scenario fall within 10 minutes. If the **Low** profile were used, the detector would not run at all (`EnablePrivilegeEscalationDetection = false`).

---

## Security Value

| Aspect | Value |
|---|---|---|
| Brute-force detection | Identifies rapid credential guessing before a password is cracked |
| Service enumeration detection | Identifies attackers probing for the weakest admin service |
| Attacker attribution | Findings include the source IP for immediate investigation |
| Time precision | Findings include exact start and end timestamps for correlation with other logs |
| Tunable response | Profile selection controls window size — defenders can tighten during incident response |
| Complementary coverage | No single attack pattern evades both sub-detectors simultaneously |

---

## Security Takeaways

1. A 12-second SSH brute-force attack with 7 attempts is immediately detected at High severity under Medium and High profiles
2. A 6-second admin service sweep across 4 ports is detected at Medium severity — a pattern that volume-based detectors would miss entirely
3. Each sub-detector catches what the other misses: spikes catch brute force, sweeps catch enumeration
4. The finding details include the specific ports targeted, giving analysts immediate context for the type of service being attacked
5. Profile-based window sizing allows defenders to tighten detection during active incident response without code changes
