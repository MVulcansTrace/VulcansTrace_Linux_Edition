# Attack Scenario: Port Scan Detection

## The Security Problem

An attacker performs a TCP SYN scan against a Linux server to identify open services before launching targeted exploits. The scan generates iptables log entries as each SYN packet hits the firewall. The port scan detector must identify this pattern among potentially thousands of legitimate connection logs and produce a finding that gives the analyst enough context to respond.

---

## Detection Walkthrough

| Step | Operation | Result |
|---|---|---|
| 1. Guard | `EnablePortScan = true`, events.Count = 16 | Proceed |
| 2. Group by source | All 16 events share `SRC=10.0.0.99` | One group |
| 3. Order by timestamp | Already sequential | 10:01:12 → 10:01:27 |
| 4. Truncate | `PortScanMaxEntriesPerSource = null` | No truncation |
| 5. Pre-window gate | 16 distinct destination ports | 16 >= 15 (Medium threshold) → proceed |
| 6. Sliding window | All events fall within a 5-minute span | One active burst |
| 7. Count distinct per window | 16 distinct destination ports | 16 >= 15 → finding triggered |

## The Finding

```json
{
  "Id": "<auto-generated GUID>",
  "Category": "PortScan",
  "Severity": "Medium",
  "SourceHost": "10.0.0.99",
  "Target": "multiple ports",
  "TimeRangeStart": "2026-01-19T10:01:12",
  "TimeRangeEnd": "2026-01-19T10:01:27",
  "ShortDescription": "Port scan detected from 10.0.0.99",
  "Details": "Detected 16 distinct destination ports within 5 minutes."
}
```

> **Note:** The `Id` field is a deterministic GUID derived from a SHA-256 hash of the finding's content fields (`Category`, `Severity`, `SourceHost`, `Target`, `TimeRangeStart`, `TimeRangeEnd`, `ShortDescription`, `Details`). Two findings with identical content produce the same Id. The year in timestamps is inferred from the system clock at parse time (`DateTime.Now.Year`), so actual output reflects the year the analysis is run. Timestamps carry `DateTimeKind.Unspecified` — they have no UTC offset.

## Design Rationale

The scan above completed in 15 seconds, targeting 16 well-known service ports (SSH, HTTP, HTTPS, HTTP-Alt, HTTPS-Alt, MySQL, PostgreSQL, SMTP, DNS, POP3, IMAP, IMAPS, POP3S, SMTPS, PPTP, RDP). This is a textbook service-identification scan — the attacker is probing for common services to identify attack surface.

The detector correctly identifies this because:

1. **All events fall within one 5-minute sliding window**, so the distinct-port count is 16
2. **16 distinct destination ports exceeds the Medium threshold of 15**
3. **The pre-window gate passes** because the total distinct count (16) is >= 15

If the same scan were run against a **Low** profile (threshold: 30), no finding would be emitted — both because 16 < 30 and because the Low profile's severity floor (`MinSeverityToShow = High`) filters out all Medium-severity findings regardless of threshold. Against a **High** profile (threshold: 8), the finding would still be emitted because 16 >= 8.

If the attacker spreads the scan so that no 5-minute sliding span contains enough distinct destination ports, the detector will not emit a finding. Under a High profile, the lower threshold makes smaller bursts easier to catch.

---

## Security Value

| Aspect | Value |
|---|---|
| Reconnaissance detection | Identifies active service mapping before exploitation begins |
| Attacker attribution | Finding includes the source IP for immediate investigation |
| Time precision | Finding includes exact start and end timestamps for correlation with other logs |
| Tunable response | Profile selection controls sensitivity — defenders can increase during incident response |
| Downstream correlation | `RiskEscalator` escalates correlated PortScan and FlagAnomaly findings to Critical if the same source generates both within the correlation window |

---

## Security Takeaways

1. A 15-second scan across 16 common service ports is detected immediately under Medium and High profiles
2. The finding provides actionable context — source IP, target count, and exact time range — enabling rapid triage
3. Time-windowed distinct-port counting is resilient to connection flooding — an attacker cannot dilute the scan by sending repeated connections to already-probed ports. However, the attacker could still evade by spreading probes beyond the configured sliding window
4. The detector's profile sensitivity allows defenders to tune detection to their network's baseline noise level
5. Downstream escalation means that if the same attacker also uses stealth scan techniques (flag anomalies), the participating PortScan and FlagAnomaly findings are escalated to Critical severity
